## WP上限内部分割方式 — 方針と課題

> 作成日: 2026-03-19

### 方針

Routes API v2 のウェイポイント上限（25WP: origin + destination + 23 intermediates）を
内部的なAPI呼び出し分割で突破し、UIとしてはWP数無制限でルート作成・ナビゲーションできるようにする。

### 分割方式

ユーザーが N 個のWPを設定した場合、25WP単位でAPI呼び出しを分割する。
前回の destination を次回の origin として連鎖させ、返ってきた legs/steps/ポリラインを結合する。

例: 60WP の場合
- Call 1: WP 1→25
- Call 2: WP 25→50（WP25 が origin）
- Call 3: WP 50→60（WP50 が origin）

### メリット

1. **ルート設定の自由度**: ツーリングルートを「作り込む」コンセプトとの親和性が高い。
   通りたい道を細かくWPで指定できるようになる。
2. **通行止め検知精度の向上**: ナビ開始時のポリライン差分比較で、
   WPが多い＝legが短いほど「どのleg内で変化が生じたか」の特定精度が上がる。
   ユーザーが「通りたい道」をWPで明示するほど、通行止め箇所が識別しやすくなる。

### ナビ開始時ポリライン差分比較（関連機能）

保存済みルートでナビを開始する際、同じWPでRoutes APIを再呼び出しし、
保存済みポリラインと比較する。leg単位で差分を検出し、変化があったlegでは：
- 保存済みポリライン（青線）と新しいポリライン（緑の破線）を重ねて表示
- ユーザーが目視で通行止め箇所を推定できる

通行止め箇所の自動特定（座標レベル）は行わない。
Google Maps API に道路単位の通行止め情報を返すAPIは存在しないため、
ポリライン差分から原因箇所を自動特定するにはleg内での二分探索が必要になり、
API呼び出し回数が爆発する。ユーザーに両方のポリラインを見せる方式で十分と判断した。

### 通行止めデータに関する調査結果

Google Maps の内部道路グラフ（ノード・エッジ）は外部非公開。
「この道路区間は通行止めか？」を直接問い合わせるAPIは存在しない。

Googleの通行止め情報ソース:
- Androidスマホの匿名位置データ（速度データがゼロになれば推測）
- 行政機関のデータフィード（日本: VICS / JARTIC）
- Waze のクラウドソーシング（ユーザー報告）
- 道路センサー（レーダー・赤外線）

外部から利用可能なデータソース:
- JARTIC オープンデータ: 無料だが月次バッチ更新（リアルタイムではない）
- VICS オンライン提供: リアルタイムだが有料・法人契約・技術開示契約が必要
- Routes API 差分比較: API呼び出し時点のリアルタイム情報を間接的に利用可能

FlexRoute では Routes API 差分比較を採用する。

### 分割ナビの既知の課題（6件）

#### 課題1: 分割点のアプローチ方向の不連続（最も深刻・原理的に解決不可）

1本のAPI呼び出し内であれば、WPへの進入方向と出発方向を同時に最適化する。
分割すると各呼び出しが独立に最適化するため、分割点（=WP）での
進入方向と出発方向が不連続になりうる。

複雑な交差点やインターチェンジ付近で顕在化する可能性がある。
ただし分割点はユーザーが設定したWPであり、行きたい場所そのもの。
大抵は道路沿いの施設やランドマークで、進入方向の差は
「右折で入るか左折で入るか」程度に収まると予想。
インターチェンジ分岐点にWPを置く極端なケースでのみ問題化する。

#### 課題2: 分割点での案内指示の重複

Call 1 の最終step「WP25に到着」と Call 2 の最初step「WP25を出発」が
重複する。1本のルートなら不要な案内。
ナビ中に「目的地に到着しました→出発します」と不自然なガイダンスが入る。

対処法: 分割点の到着/出発stepをフィルタリングで除去する。

#### 課題3: 分割呼び出し間の所要時間の不整合

複数回のAPI呼び出しはタイミングがずれるため、
TRAFFIC_AWARE の渋滞情報に差が出る。
合計所要時間が1本で呼んだ場合と一致しない。

対処法: 各呼び出しの所要時間を単純合算する（厳密な整合性は諦める）。

#### 課題4: ポリラインの微小ギャップ

Call 1 の最終座標と Call 2 の最初の座標は同じWPだが、
道路へのスナップ処理が独立しているため数メートルのズレが生じうる。
地図上でポリラインが途切れて見える可能性がある。

対処法: 結合時に座標を補間する。

#### 課題5: リルート時の呼び出し範囲

ナビ中の逸脱→リルートで、分割境界をまたぐ場合に
どこからどこまで再計算するかの管理が複雑になる。

対処法: リルート時は「現在のleg内のWP」から最終目的地までを再分割して呼び出す。

#### 課題6: サムネイル・距離・時間の集計

複数レスポンスの結合で1つの encodedPolyline、totalDistanceMeters、
totalDurationSeconds を作る必要がある。

対処法: 結合ロジックの実装（技術的に解決可能）。
