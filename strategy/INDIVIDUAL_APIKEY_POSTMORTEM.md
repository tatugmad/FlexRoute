# 個人APIキー戦略 — 検討経緯と断念理由
> 最終更新: 2026-03-16
> **重要な失敗記録。同じ検討を繰り返さないために残す。**
## 戦略の概要
FlexRoute ユーザー個人の Google Cloud アカウントに API キーを自動作成し、
そのキーで FlexRoute が動作する方式。
ユーザー個人の無料枠を活用し、運営の API コストをゼロに抑える狙い。
## 検討の経緯
### 出発点
1ユーザーの月間利用は全 SKU で無料枠の6〜20%しか消費しない。
もし各ユーザーが自分の無料枠を使えれば、理論上は何ユーザーでも API コストゼロ。
### 検討した方式
#### 方式A: OAuth でプロジェクト自動作成
1. ユーザーが「Google でログイン」
2. OAuth トークンで FlexRoute バックエンドが:
   - Cloud Resource Manager API でプロジェクト作成
   - Service Usage API で Maps API 群を有効化
   - API Keys API でキーを発行（リファラ制限付き）
   - Cloud Billing API で請求先アカウントをリンク
3. ユーザーは API キーの存在を意識しない
#### 方式B: ユーザーに手動でキーを発行させる
セットアップガイド付きで Google Cloud Console の操作を案内。
#### 方式C: FlexRoute のカードでユーザーのプロジェクトの請求先を登録
ユーザーのカード不要。FlexRoute が全ユーザーの請求を負担。
## 断念理由1: 規約違反（致命的）
Google Maps Platform Terms of Service, Section 3.2.1(c)(ii):
> "Customer will not access or use the Services in a manner intended to avoid incurring Fees"
ユーザーごとに請求先アカウントを分離して個別の無料枠を活用する戦略は、
「料金回避を意図した利用方式」に該当する。
Google の裁量でアカウント停止される可能性があり、
FlexRoute 全体のサービスが突然停止するリスクがある。
## 断念理由2: 技術的に不可能（致命的）
### 請求先アカウントの作成と支払い方法の登録は API で実行できない
Cloud Billing API でできること:
- ✅ 既存の請求先アカウントの一覧取得
- ✅ 請求先アカウントとプロジェクトのリンク
- ✅ サブアカウント作成（リセラーのみ）
- ✅ 予算アラートの設定
Cloud Billing API でできないこと:
- ❌ **請求先アカウントの新規作成**（Google Cloud Console でのみ可能）
- ❌ **クレジットカードの登録**（Google Cloud Console でのみ可能）
- ❌ **Google Payments プロファイルの作成**（手動のみ）
つまり、FlexRoute のアプリ内で完結するシームレスな体験は実現できない。
ユーザーは必ず Google Cloud Console に遷移して手動操作が必要。
### 手動操作のハードル
Google Cloud Console での請求先アカウント作成は以下のステップを要する:
1. Google Cloud Console にアクセス
2. 請求先アカウントの作成ウィザードを開始
3. 国の選択、利用規約への同意
4. クレジットカード情報の入力
5. 作成完了
一般のバイクツーリング愛好家にこの操作を求めるのは非現実的。
## 関連する事実
### Google アカウントのカード登録状況
- Google アカウント保持者の約40〜50%が Google Pay / Google Play に支払い方法を登録済み
- ただしこれは Google Cloud Platform の請求先登録とは別物
- Google Cloud に請求先を登録しているのは開発者・企業に限られ、一般ユーザーの1%未満
### 無料枠の適用単位
新制度（2025年3月〜）では SKU ごとの無料回数が請求先アカウント単位で適用。
複数プロジェクトを同一請求先にリンクしても無料枠は1セットのみ。
別の請求先アカウントを作れば別の無料枠が得られるが、上記の規約違反に該当。
## 教訓
1. **Google の規約は「意図」で判断される。** 技術的に可能でも、料金回避の意図が明確な利用方式は禁止される。
2. **Google Cloud の課金インフラは API 化されていない。** プロジェクト作成や API 有効化は自動化できても、請求先の作成と支払い方法の登録は手動のみ。これは Google のセキュリティポリシーと考えられる。
3. **Google Maps API を活用したアプリが少ない根本原因はこのコスト構造にある。** 個人向けサービスで従量課金 API のコストを回収するのは困難。この壁を突破するアイデアが見つかれば画期的なアプリになる。
